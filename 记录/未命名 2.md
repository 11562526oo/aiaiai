1. **1亿热数据sql表怎么存，全是热数据如何拆分**
步骤1：存储层核心设计（单表优化）
目的：保障单表在高并发读写下的性能基线。
操作：
存储引擎选 InnoDB（支持事务、行级锁、MVCC，适配热数据的并发场景）；
字段设计极简且精准（避免冗余，减少IO与内存开销）；
索引优化：对查询/排序高频字段建索引（区分度高的列优先，避免过度索引拖慢写入）；
冷热分离兜底：若未来数据有冷热分层，提前预留分区/分表规则（但当前全热，暂聚焦单表性能）。
步骤2：全热数据拆分策略（水平/垂直拆分）
目的：突破单表容量与性能瓶颈，分散压力到多节点/表。
操作（优先水平拆分，因全热无业务模块隔离）：
按ID哈希分表：如 user_id % 1024分成1024张表，分散写入与查询压力；
按时间分区：若数据带时间属性（如订单、日志），用数据库原生分区（如MySQL RANGE分区）按天/月拆分；
读写分离配合：主库负责写+实时读，从库负责延迟读，通过中间件（如MyCat、ShardingSphere）路由请求。


2. **redis zset 的 Sort member 哪个不能重复为什么**
member不能重复，score可重复。
原因：
Redis zset（有序集合）的底层实现是「跳表 + 哈希表」：
哈希表负责 member到 score的映射，保证 member唯一（重复插入相同 member会覆盖旧 score）；
跳表基于 score排序，允许多个 member共享相同 score（此时按 member字典序排序）。
这种设计既满足了“按分数排序”的需求，又通过 member唯一性实现了集合的去重特性。


3. **数据库1w个数据(字段a)分数不同 高访问量下前端传给你100个字段a数据 你能想到的最快排序的返回方式**
步骤1：缓存预热（提前准备有序数据）​
把数据库中所有“字段a → 对应数据”的关联关系，以 Redis zset​ 存储（zset天然按分数排序）；
示例：ZADD sorted_a  score1 member1  score2 member2 ...（member为字段a的唯一标识，score为分数）。
步骤2：前端请求时直接读缓存​
前端传100个字段a的标识后，通过 ZRANGE或 ZMSCORE等命令，批量从Redis zset中获取有序数据；
若缓存未命中（如首次加载），则触发“数据库查询 + 缓存更新”逻辑（需控制并发，避免击穿数据库）。

4. **缓存10w个数据，如果用hash结构，怎么存，怎么拿**
步骤1：Hash分片（拆分存储单元）​
因Redis单个Hash不宜过大（建议<1000字段，或控制内存占用），需按规则拆分到多个Hash：
规则示例：按数据ID取模（如 data_id % 100），分成100个Hash，命名为 hash:data_0, hash:data_1...；
每个Hash内，以“数据字段名”为field，“数据值”为value（如 hset hash:data_0 name "Alice" age 25）。
步骤2：批量写入优化​
用 pipeline批量执行 hset（减少网络IO次数），或借助Redis集群的分片特性分散压力。
获取逻辑：精准Key + 批量读取​
单条获取：HGET hash:data_{n} field（如 HGET hash:data_0 name）；
多条获取：HMGET hash:data_{n} field1 field2 ...（批量取一个Hash内的多个字段）；
跨Hash聚合：若需跨分片取数据，先通过ID定位Hash，再批量读取（逻辑由业务层封装）。
