G1：低延迟和高吞吐量
	- G1 将堆内存划分为多个**相等大小的区域（Region）**，而不是像其他垃圾回收器那样简单地将堆分为固定大小的新生代、老年代。
- 每个 Region 都可以动态地作为 Eden 区、Survivor 区或 Old 区，这种灵活的内存划分方式让 G1 更加高效。

	**提供可预测的暂停时间**  **`-XX:MaxGCPauseMillis`**


|**垃圾回收器**|**特点**|**缺点**|
|---|---|---|
|**Serial GC**|单线程回收，适合单核 CPU，低内存场景。|只使用一个线程进行 GC，无法利用多核 CPU，GC 停顿时间长。|
|**Parallel GC**|多线程回收，适合高吞吐量场景。|停顿时间较长，不适合低延迟场景。|
|**CMS GC**|并发回收，低延迟，适合响应时间要求高的场景。|容易产生内存碎片；在高负载下可能触发 Full GC，导致长时间停顿。|
|**G1 GC**|分区管理，低延迟，可控暂停时间，自动压缩内存，适合大内存场景。|对吞吐量的优化不如 Parallel GC；在小内存场景下性能可能不如 CMS。|
|**ZGC**（Java 11+）|超低延迟（10ms 以下），支持超大堆内存（TB 级别）。|仅支持 Java 11 及以上版本，且对 CPU 资源需求较高。|
|**Shenandoah GC**|超低延迟，垃圾回收时间与堆大小无关，适合大内存和低延迟场景。|需要 Java 12 及以上版本，且对 CPU 资源需求较高。|
