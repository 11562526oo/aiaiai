




**业务要求每5分钟为下游系统提供一份完整的数据。**
数据包括所有企业、企业下订购情况、以及每个订购下的所有域名。主要是从两个主表（企业表和域名表）JOIN查询，还有一些其他的小表中查询，最终返回的数据量达到270多万条记录。

虽然日均访问只有几万次，但大数据和下游期望查询能快速返回，不能等待十几秒钟。

1. 首先因为数据量过多，优化体验

2. 要考虑5分钟粒度，定时任务5分钟返回所有的最新数据。
	ComplatableFuture并发查询，优化sql，使用流式处理序列化
	如果有小于一定数量的key更新失败，会被记录，仍然更新，失败的会被记录在下一个定时中更新

3. 大数据量和缓存的冲突(大key问题)
	采用了分段存储方案，不是把所有的存在一个key，而是按照企业维度分段。每个key的体积控制在10Mb以内

实现步骤：
	1.后台开启一个5分钟查询的定时任务。按照企业维度进行分组和序列化
	2.为了防止多段key中有的key更新失败，采用版本化的方案，写入临时的keyv2，只有所有的key都写入成功，才会原子性的更新为v2。如果有更新失败则发出告警邮件，并仍使用v1。
	3.查询会先读取版本指针，根据需要的企业列表分段读key。

查询延迟降低，数据库压力减缓，系统的稳定性增强，并发查询也不hi对数据库造成压力。

4. 5分钟任务更新周期可控性
	使用分布式锁 + 手动告警
